Quantitative Aptitude

1 → (b) ₹2100. CI = amount − principal. Amount = 10000 × 1.1 × 1.1 = 10000 × 1.21 = 12100. CI = 12100 − 10000 = 2100. Trick: for 2 years, CI% = a + a + a²/100 = 10+10+1 = 21%.

2 → (b) 28%. Successive-change formula: a + b + ab/100 with both negative → −20 −10 + (−20)(−10)/100 = −30 + 2 = −28% net, i.e. a single 28% discount. Common trap: answering 30% (just adding).

3 → (b) 5 days. LCM method: work = LCM(10,15,30) = 30 units. Rates: A=3, B=2, C=1 per day → together 6/day → 30/6 = 5 days. No fractions needed if you pick LCM units.

4 → (b) 40 km/h. Equal distances → harmonic mean: 2xy/(x+y) = 2×30×60/90 = 40. Never (30+60)/2 = 45 — that's the trap option.

5 → (b) 40. Let son = s, father = 4s. In 20 yrs: 4s+20 = 2(s+20) → 4s+20 = 2s+40 → 2s = 20 → s=10, father = 40. Translate English to equation immediately; don't juggle in your head.

6 → (a) 7:5. 40 L at 7:3 → milk 28 L, water 12 L. Add 8 L water → 28:20 → divide by 4 → 7:5. Only the water changes; milk stays 28.

7 → (b) 9240. Multipliers: 8000 × 1.05 × 1.10 = 8400 × 1.10 = 9240. Same multiplier chain you use for profit questions.

8 → (b) 2 km/h. Downstream speed = 20/2 = 10; upstream = 12/2 = 6. Stream = (down − up)/2 = (10−6)/2 = 2. Boat speed would be (10+6)/2 = 8.

Logical Reasoning

9 → (b) 159. Each term = previous × 2 + 1: 4→9→19→39→79→159. When differences look messy, test ×2±k next.

10 → (c) 9. 3, 5, 7, 11 are prime; 9 = 3×3 is not.

11 → (a) MBQUPQ. MOBILE→NPCJMF is each letter +1. LAPTOP: L→M, A→B, P→Q, T→U, O→P, P→Q → MBQUPQ. Write A=1…Z=26 on rough paper first — 30 seconds well spent.

12 → (b) 5 km. North 4 and east 3 are perpendicular → distance = √(4²+3²) = 5. The classic 3-4-5 triangle. Direction from start: north-east.

13 → (a) only I. "All roses are flowers" guarantees some flowers are roses (I true). II fails: the red flowers might all be non-roses — draw the Venn: red circle can overlap flowers without touching roses. A conclusion must be true in EVERY possible diagram.

14 → (b) Saturday. 61 mod 7 = 5 (since 56 is divisible by 7). Monday + 5 days = Saturday. Always mod 7 first, never count 61 days.

Code Output

15 → (b) 1 2 3. static variables initialize once and persist across calls — c survives between calls, incrementing each time. A normal local would print 1 1 1.

16 → (b) 12. *(a+1) = a[1] = 4 and *(a+3) = a[3] = 8 → 4+8 = 12. Pointer arithmetic moves by whole elements, not bytes.

17 → (c) C. 'A' is 65; 65+2 = 67 = 'C'. The %c format prints it as a character. With %d it would print 67 — watch the specifier.

18 → (b) 5 10. Java is strictly pass-by-value: swap gets copies of a and b; swapping copies changes nothing outside. Yes, even objects — the reference itself is passed by value.

19 → (b) Child's. Overriding + a Parent reference to a Child object → the runtime type (Child) decides which method runs. That's runtime polymorphism / dynamic dispatch.

20 → (b) 5. i --> 0 is just (i--) > 0 written cheekily ("goes-to operator" joke). Checks i>0 with old value, then decrements: prints 4 3 2 1 0 — five numbers.

DBMS

21 → (b). An index is a lookup structure (usually B-tree) that makes SELECTs fast, but every INSERT/UPDATE must also update the index — writes get slower. Classic trade-off question.

22 → (c) Isolation. A=all-or-nothing, C=valid state to valid state, I=concurrent transactions don't see each other's half-done work, D=committed data survives crashes.

23 → (b). LEFT JOIN keeps every row of the left table; where the right side has no match, its columns come back NULL. That's exactly why it's used for "including zero" counts (see C2).

24 → (b). A table can have several candidate keys (all unique+not-null capable); the primary key is the one you choose. Auto-increment is a convenience, not a requirement.

OS, Networks & Shell

25 → (c) compiled. Process states are about execution: new, ready, running, waiting, terminated. Compilation happens before a process ever exists.

26 → (b). Virtual memory lets programs use more memory than physical RAM by paging to disk — each process sees its own large address space.

27 → (b). A mutex is a lock with ownership (the locker must unlock, binary). A semaphore is a counter — can allow N holders and be signalled by a different thread. One-liner: mutex = key to one toilet; semaphore = N parking slots.

28 → (c) 443. HTTP=80, HTTPS=443, SSH=22, DNS=53, SMTP=25. Memorize these five.

29 → (b) Layer 3. IP addressing/routing = Network layer. L2=switches/MAC, L4=TCP/UDP, L7=HTTP.

30 → (b) wc -l. "word count" with -l counts lines. grep -n shows line numbers of matches; cat -n numbers output; ls -l is long listing.

Section B — Pseudocode

B1 → 32 31. Trace a table of (x, y) per loop pass:
start x=1,y=0 → pass 1: y=0+1=1, x=2 → pass 2: y=1+2=3, x=4 → pass 3: y=3+4=7, x=8 → pass 4: y=7+8=15, x=16 → pass 5 (16<20 still true): y=15+16=31, x=32 → now 32<20 fails, loop ends. Print x then y: 32 31. Trap: forgetting the pass where x=16 (it's still <20), or printing y before x.

B2 → 8. This is Fibonacci: g(0)=0, g(1)=1, then each = sum of previous two. Build up, don't recurse in your head: 0, 1, 1, 2, 3, 5, 8 → g(6) = 8. Whenever you see g(n-1)+g(n-2), write the sequence forward — far faster and safer than tracing the recursion tree.

Section C — SQL

C1 — the key idea is a correlated subquery: the inner query re-computes the average for each employee's own department because it references the outer row (e.dept_id).

SELECT e.name
FROM employees e
WHERE e.salary > (SELECT AVG(salary) FROM employees
                  WHERE dept_id = e.dept_id);

Why not a plain subquery? WHERE salary > (SELECT AVG(salary) FROM employees) compares against the company-wide average — wrong question. The correlation (WHERE dept_id = e.dept_id) is what scopes it per department. If asked for an alternative: JOIN against a derived table of per-dept averages — same result, often faster on big tables.

C2 — two ideas working together: ① LEFT JOIN keeps departments with no employees (their employee columns come back NULL); ② COUNT(e.emp_id) counts only non-NULL values, so empty departments correctly show 0.

SELECT d.dept_name, COUNT(e.emp_id) AS emp_count
FROM departments d
LEFT JOIN employees e ON e.dept_id = d.dept_id
GROUP BY d.dept_name;

Trap: COUNT(*) would return 1 for an empty department (it counts the row itself, NULLs and all). And an INNER JOIN would drop empty departments entirely. This exact pattern — "including zero" — is a panel favorite.

Section D — Coding

D1 — Move Zeros (two-pointer "writer" technique), O(n) time, O(1) space:

def move_zeros(a):
    w = 0                       # w = where the next non-zero should go
    for i in range(len(a)):     # i = reader, scans everything
        if a[i] != 0:
            a[w], a[i] = a[i], a[w]   # move non-zero forward
            w += 1
    return a

How it works: w only advances when a non-zero is placed, so everything before w is the non-zeros in original order, and the zeros naturally end up swapped to the back. Dry-run [0,1,0,3,12]: i=1 swaps 1 to front → [1,0,0,3,12]; i=3 swaps 3 → [1,3,0,0,12]; i=4 swaps 12 → [1,3,12,0,0]. Edge cases to mention: all zeros (loop does nothing harmful), no zeros (every element swaps with itself), empty array. The naive alternative — building a new list — is O(n) extra space; say why you avoided it.

D2 — Anagram Check, O(n) time:

from collections import Counter
def is_anagram(s, t):
    return Counter(s) == Counter(t)

Counter builds a frequency map of each string in O(n); equal maps = same letters with same counts = anagram. Mention the guard len(s) != len(t) → False as a fast exit. If asked "without libraries": use a dict (or a 26-slot array for lowercase), +1 for s, −1 for t, check all zeros. The sort-both-strings alternative works but is O(n log n) — say you chose the counter for linear time; interviewers ask complexity here, which is why the question includes it.

Section E — Design Theory

Best answer: both, split by responsibility. Model answer with the reasoning spelled out:

① Bookings must be SQL. "Never double-book a room" is a textbook ACID requirement: the check-availability + reserve steps must be one atomic transaction, and row-level locking / unique constraints make two simultaneous bookings of the same room impossible. Eventual consistency here = two families at one door.

② The catalog fits NoSQL (or MySQL JSON columns). Hotels having wildly different amenities/room types/seasonal pricing is a schema-flexibility problem — a document per hotel absorbs the variety without endless nullable columns or table alterations.

③ Read-heavy traffic → cache + replicas. Listings are read thousands of times per booking, so put Redis (or a CDN for images) in front and scale reads with replicas; the write-critical booking path stays small and strongly consistent.

Why this scores: the question is engineered to look like "pick one," but the mature answer is matching guarantees to requirements — consistency where money/inventory moves, flexibility where content varies, caching where reads dominate. If forced to choose exactly one: choose SQL and say the catalog flexibility can be handled with JSON columns — correctness of bookings outranks catalog convenience. (Bonus: you've effectively used this split in A.R.G.U.S-V — Redis cache in front of expensive computation — mention it.)
